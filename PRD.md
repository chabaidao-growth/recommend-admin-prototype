# 推荐系统运营管理后台 PRD

> **版本：v3.4 | 更新日期：2026-05-15**

> 本文档描述茶百道小程序点单页推荐系统运营管理后台的产品功能、配置能力与底层业务逻辑。  
> 适用于产品、研发、运营、装修、数据团队对齐实现方案。  
> 文档风格以"产品需求 + 业务逻辑 + 操作逻辑"为主，**接口字段、字典枚举、缓存口径由研发自行约定**；不可绕过的规则集中以 **【关键业务规则】 / 【关键操作规则】** 标出。

---

## 1. 产品概述

推荐系统运营管理后台覆盖**三层架构**的配置能力，每层之间逐层引用：

```text
策略组合 → 排序策略 → 选品池 → 商品
```

运营在后台完成从"商品候选"到"策略编排"再到"展示组合"的配置。  
**展示触发由装修系统**：装修在自己侧维护「资源位 → 组合 ID」映射，按组合 ID 调用本系统（详见 5.3 / 5.4 / 6 章）；本后台不维护资源位，也不再承担"投放计划"层。

---

## 2. 架构总览

### 2.1 三层架构关系

| 层级 | 实体 | 作用 | 关键问题 |
|---|---|---|---|
| 第一层：选品池 | Pool | 定义"哪些商品可以被推荐" | 候选范围划定 |
| 第二层：排序策略 | Strategy | 定义"候选商品如何排序" | 排序规则与兜底链 |
| 第三层：策略组合 | Combination | 定义"N 个坑位（1-50）分别由哪个排序策略产出，以及展示顺序" | 坑位编排与对装修的稳定 ID |

### 2.2 实体业务含义

- **商品（Product）**：来自实时数仓清洗的商品名称（**非 SKU 非 SPU**）；数仓原始数据含饮品 / 套餐 / 小料等所有商品类型，但**选品池仅承载饮品**，其他品类不进入选品池；详见 3.1 与 3.2
- **选品池（Pool）**：**仅饮品集合**，分为"系统内置全量池（仅饮品）"和"自定义池（仅饮品）"两类；选品池的添加、导入、搜索均仅在饮品范围内进行
- **排序策略（Strategy）**：选定一种排序模式 + 关联一个选品池（人工定序除外），决定候选如何排序
- **策略组合（Combination）**：自定义坑位数（slotCount，1-50）+ 每个坑位绑定一个排序策略；**不绑定资源位**，资源位由装修系统在自己侧映射

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **逐层引用、单向依赖**：上层引用下层，下层不感知上层
- **删除前的反向引用校验有差异**：
  - **选品池被策略引用** → 禁止删除
  - **策略被组合引用** → 禁止删除
  - **组合**：本后台不强制校验装修引用（无法实时反查），允许删除；详见 5.8.5
- **状态变化沿链路传播**：停用下层后，上层运行时走兜底，不自动解绑引用，便于运营恢复
- **装修系统是唯一消费方**：组合 ID 作为装修系统配置的稳定外部 ID（详见 5.4），本后台不再承载"投放计划 / 优先级 / 冲突检测 / ABTest"，这些能力由装修系统负责
- **业务对象的编辑权限仅授予创建人**：组合 / 策略 / 池的编辑权限由创建人独占，不被普通角色权限覆盖（**超管除外**，详见 5.8.2 / 9.3）

</callout>

## 3. 第一层：商品选品池

### 3.1 商品定义与范围

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **推荐对象是"商品名称"，不是 SKU 也不是 SPU**：因为推荐展示给消费者看的就是商品名（如"豆乳玉麒麟"）
- **数仓原始数据含全品类**：饮品 / 套餐 / 小料 等都在数仓中
- **选品池及 HOT / NEW / ALGORITHM 只保留饮品**：选品池的可视范围限定为"饮品类"，自定义池的添加 / 批量导入 / 搜索均只搜饮品；HOT / NEW / ALGORITHM 三种排序模式只消费选品池里的饮品
- **池可附带 SPU ID 用于校验**：选品池在内部可保存商品对应的 SPU ID（来源于第一张表的去重键），仅用于"导入校验 / 同名核对"等底层校验场景；**对外暴露与算法输出仍以"商品名称"为单位**，UI 列表不展示 spuId
- **MANUAL 例外（不依赖选品池）**：MANUAL 模式直接从"数仓全量商品"中挑选（含饮品 / 套餐 / 小料），见 4.4

</callout>


### 3.2 数据来源

数仓侧提供 **3 张底表** 供推荐系统消费：

| 底表 | 业务含义 | 关键字段 | 时效 | 消费方 |
|---|---|---|---|---|
| 第一张表 — **原数据 / 在售商品** | SPU 维度聚合全国饮品（含套餐等全品类原始数据）；**只要一家门店在售即算入**；含商品名称、SPU ID、品类、上架时间、下架状态 | 商品名称 / SPU ID / 品类 / 上架时间 / 在售状态 | 准实时 | 选品池商品基底；MANUAL 全量商品清单 |
| 第二张表 — **销量统计** | SPU 维度聚合饮品销量；指标 = 杯数 + GMV；维度 = 门店 + 全国；时间 = 7 / 14 / 30 天滚动；**T+1 离线** | SPU ID / 商品名称 / 杯数 / GMV / 维度（门店/全国）/ 时间窗口 | **T+1 离线** | HOT 热销排行 |
| 第三张表 — **新品排行** | 在第一张表基础上扩展字段；以"近 7 天上新时间"为排序键；自动过滤已非新品 | 商品名称 / 上新时间（近 7 天） | 准实时 | NEW 新品排行 |
| （超大表）| 用户行为日志 + 偏好画像 | 用户行为序列、特征向量等 | 实时 / 准实时 | ALGORITHM 算法模型 |

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **SPU 用于去重，名称用于推荐**：底层用 SPU ID 去重，对外仍以"商品名称"作为推荐对象；一个 SPU 可能对应多个地区名（如"豆乳玉麒麟"/"豆乳麒麟"），运营在底表中按需归一
- **上下架由数仓维护**：商品下架时数仓自动将状态置为"下架"，本后台不感知；运行时由排序策略消费数仓字段实现"自动过滤"；下架商品在第一张表内**保留**，运行时过滤
- **重新上架自动恢复**：商品重新上架后状态自动回到"在售"，无需运营手动操作
- **销量数据 T+1 不实时**：第二张表的销量指标按 T+1 离线产出，运营对效果监控的实时性预期需相应调整
- **数仓暂无 SPUID 特征向量**：ALGORITHM 当前仅以"商品名称"作为特征键；如未来引入 SPU 特征，需算法侧专门建模

</callout>


### 3.3 系统内置池 vs 自定义池

| 类型 | 标识 | 创建 | 增删商品 | 删除池 |
|---|---|---|---|---|
| 系统内置池（ALL 全量池）| `SYSTEM` | 系统硬编码 | 不允许手动 | 不允许 |
| 自定义池 | `CUSTOM` | 运营创建 | 允许 | 允许（需无策略引用）|

- **新建选品池**：表单输入名称（≤30 字）、描述（≤200 字）、初始状态
- **查看列表**：卡片式展示；**排序口径 = 系统内置池置顶 + 其余按创建时间倒序（新→旧）**
- **查看详情**：基础信息（可编辑）+ 商品清单（**商品量大时分页**，默认每页 50 / 100 / 200 可调）+ 关联策略引用
- **添加商品**（仅自定义池）：
  - 弹窗内**实时从数仓饮品全量商品**搜索；非饮品不会出现在搜索结果
  - **复选框多选 + 一键批量添加**（不再逐个 confirm）
  - **保存时去重**：与池中已有商品取并集，重复名称不会重复入库
- **新建池"全选"**：在添加商品弹窗中提供"全选当前结果"按钮；**但不支持"动态全量池"** —— 全选只把当下勾选结果写入快照，**不会订阅后续新品自动入池**
- **批量导入**：**仅支持文本框粘贴商品名称**（**换行 / 英文逗号** 分隔，上限 500），不支持附件上传；导入时校验仅命中饮品，非饮品视为无效
- **移除商品**：单个或批量（仅自定义池）
- **启用 / 停用**：停用后引用该池的策略走兜底
- **删除**：仅自定义池且无策略引用；**必须先停用才能删除**（启用态时删除按钮置灰，Tooltip 提示"请先停用后再删除"）；删除时输入池名二次确认

<callout emoji="📌" background-color="light-yellow">

**【关键操作规则】 — 选品池**

- **添加商品交互**：勾选 → 一键添加 → 保存去重；不是"勾一个确认一个"
- **不支持动态全量池**：所有"全选"都是当前时点的快照，新增饮品需运营手动加入或重做一次"全选"
- **超管可越权管理别人创建的自定义池**（编辑 / 停用 / 删除）；其他角色仅能管理自己创建的池

</callout>

### 3.5 批量导入校验

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **校验由后端完成**，前端只负责录入与展示
- **分隔符**：换行 / 英文逗号；其它符号（如中文逗号、空格）视为同一段名称内容
- **三类结果分类展示**：有效商品 / 已重复 / 无效名称
- **预览后确认**：用户在预览面板看到分类后再点击"确认导入"才真正写入
- **校验对接数仓**：以"商品名称"为键查找是否存在、在售、且属于饮品；非饮品视为"无效名称"

</callout>

## 4. 第二层：排序策略

### 4.1 功能说明

排序策略定义**"候选商品如何排序"**。每个策略选择**一种排序模式**，并按模式约定关联或不关联选品池。每个策略还需配置一个**业务标签**和一张**展示图片**用于在投放后台的策略列表 / 详情中识别。

**策略元信息字段**：

| 字段 | 业务含义 |
|---|---|
| 策略名 | 运营自定义中文名，1-30 字，保存时必填 |
| 业务标签 | 单选业务标签（如"新人专享"/"会员专享"/"日常推荐"/"活动专题"等），**必填**，标签集合由运营初始化维护 |
| 展示图片 | 单张图片，**必填**；格式 JPG / PNG / GIF；大小 **≤ 100 KB**；上传前预览，超过限制阻止上传并 Toast 提示 |
| 排序模式 | 四选一（详见 4.2）|

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】 — 标签与图片**

- **标签与图片均为必填**：保存时校验，缺失则阻断保存
- **图片格式 & 大小校验**：仅允许 JPG / PNG / GIF；体积 ≤ 100 KB；前端在选择文件时实时校验并提示"图片大小不能超过 100KB / 仅支持 JPG/PNG/GIF 格式"
- **标签词表由运营维护**：本 PRD 不限定标签内容；列表页支持按标签筛选

</callout>


### 4.2 四种排序模式

| 模式 | 业务逻辑 | 是否关联选品池 | 数据源 | 适用场景 |
|---|---|---|---|---|
| 热销排行 `HOT` | 按销量 / GMV 排序，可选维度 / 数据范围 / 时间窗口 | 是 | 数仓表 2 | 日常运营、主推爆款 |
| 新品排行 `NEW` | 按上架时间排序 | 是 | 数仓表 3 | 新品推广期 |
| 人工定序 `MANUAL` | 运营从全量商品手动拖拽排序，所见即所得 | **否（解绑选品池）** | 无表，运营自定义 | 活动、专题 |
| 算法模型 `ALGORITHM` | 基于用户特征与偏好画像的智能排序 | 是 | 数仓超大表 | 千人千面、个性化 |

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **模式四选一互斥**：一个策略只能选一种排序模式
- **创建后不可切换模式**：避免配置混乱；如需更换排序方式，请新建策略
- **过滤不可售强制开启**：所有策略统一过滤已下架商品，不暴露开关
- **MANUAL 不绑池**：人工定序的商品集合由运营在策略内自由选定（含套餐、小料等非饮品类）；其余模式仍需绑定选品池
- **统一热销兜底**：非系统策略的兜底策略固定为系统热销排行；系统策略本身不再兜底

</callout>


### 4.3 新品排行（NEW）细则

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **按商品名称聚合**：推荐系统以"商品名称"为推荐对象，同名不同 ID 视为同一商品
- **上架时间取最新值**：对同一商品名称下所有商品 ID 关联的"上架时间 / 货架时间 / 产研添加时间"取**最大值**作为该商品名称的"最新上架时间"
- **时间窗口可配**：默认 7 / 14 / 30 天，由运营选择

</callout>


### 4.4 人工定序（MANUAL）细则

- 运营**不需要选择关联选品池**
- 在策略编辑页直接从**数仓全量商品列表**勾选商品并拖拽排序
- 可选品类：饮品 + 套餐 + 小料（全品类）
- 顺序数组即为最终展示顺序
- 全量商品清单中未被勾选的商品不参与该策略
- **手动选品数量无上限**：运营可勾选任意数量商品

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **MANUAL 是唯一可挑非饮品的入口**：业务上人工定序常用于推套餐、组合促销等
- **手动维护、所见即所得**：不依赖数仓表
- **下架商品在运行时自动跳过（不占坑位）**：取数时遇下架商品继续从 MANUAL 序列取下一个有效商品；仅当**全部手动选品均下架或耗尽**时，该坑位才进入兜底策略

</callout>


### 4.5 算法模型（ALGORITHM）细则

- 推荐结果由后端推荐服务计算
- 用户行为特征、偏好画像由数据团队维护
- 后台只能"选用"该模式，不暴露模型参数

### 4.6 人工加权（进阶功能）

> 默认关闭、按需开启。

**加权计算**：
- 排序得分 α = （总数 - 原始排名 + 1）/ 总数，**归一化后数值越大排名越靠前**（取值范围 (0, 1]）
- 加权系数 β ∈ [-1, 1]，**正值提权（数值越大越靠前）**，负值降权
- 最终得分 = α + β，**按最终得分降序输出**重排后的排序

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **MANUAL 不支持加权**：人工定序顺序已固定，加权无意义；仅 HOT / NEW / ALGORITHM 三种模式可开
- **加权不改变兜底**：加权只影响主策略结果，兜底链不参与加权
- **加权配置可逐商品维护**：运营选定具体商品并指定 β 值

</callout>


### 4.7 兜底策略链

当主策略候选耗尽（如池为空、商品全过滤）时，按下述流程兜底：

1. 主策略返回候选不足 → 沿"兜底策略"取剩余坑位
2. 兜底默认指向系统热销排行（ALL 全量池 / **门店** / 近 7 天 GMV）
3. 兜底策略再不足时，**跳过坑位**（坑位空置），数据上报"坑位空置"

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **兜底数据范围 = 门店级（非全国）**：兜底策略的销量统计口径固定为"当前请求门店"近 7 天 GMV。**原因**：装修服务调用推荐时传入的"门店可售商品列表"是权威候选，全国销量靠前的商品在该门店未必有货，全国口径会导致大量兜底命中商品在门店缺货被过滤
- **跨策略全局去重**：同一推荐请求中已展示过的商品，兜底取下一个，不重复推荐
- **兜底不递归**：兜底策略自身候选不足时不再向下兜底；交由前端表现层"空置"

</callout>


### 4.8 操作能力

- **新建 / 编辑 / 启停 / 删除**：流程与所有者守则同 5.8
- **列表展示**：策略名、标签、缩略图、模式、关联池（MANUAL 显示"—"）、状态、更新时间
- **详情页**展示策略基本信息、标签、图片、模式参数、关联组合反查

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】 — 删除前必须先停用**

- **状态为「启用」时「删除」按钮置灰**，悬停 Tooltip 提示「请先停用后再删除」
- 与组合 5.8.5 删除规则保持一致：避免在线策略被误删导致引用组合在运行时落入兜底

</callout>

## 5. 第三层：策略组合

### 5.1 功能说明

策略组合（Combination）定义**"N 个坑位（Slot）分别由哪个排序策略产出商品，以及展示顺序"**，是供装修系统直接调用的"成品推荐配置"。

- 一个策略组合定义自己的**坑位数**（slotCount，1-50，由运营在创建时设定）
- 每个坑位绑定一个排序策略，推荐服务按坑位顺序逐位产出商品
- **资源位概念不在本后台维护**：装修系统在自己侧做「资源位 → 组合 ID」的映射，本后台对资源位无感
- 装修系统按「组合 ID」调用本后台获取推荐结果

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **本后台不维护资源位**：组合无资源位字段，装修系统自行处理"资源位 → 组合 ID"映射
- **坑位数由运营自定义**：创建组合时填写 1-50 之间的整数，保存后不可改动（如需变更走"复制 + 新建"）
- **坑位顺序即展示顺序**：slots 数组顺序 = POS 1 / POS 2 / … 顺序，调整顺序即调整展示位置
- **组合 ID 创建后不可变、不复用**：删除某个组合后，该 ID 不再被新建占用，确保装修配置的稳定引用
- **保存三层校验**：组合名称 + 坑位数（1-50）+ 每个 POS 的排序策略，三者均合规才能保存

</callout>


### 5.2 核心字段与业务含义

**策略组合（Combination）**

| 字段 | 业务含义 |
|---|---|
| 组合 ID | 由系统自动分配的唯一标识，**创建后不可变、不复用**，用于装修配置稳定引用（详见 5.4）|
| 组合名称 | 运营自定义的中文名，1-30 字，保存时必填 |
| 状态 | 启用 / 停用 |
| 坑位数（slotCount）| 运营自定义的整数，**取值范围 1-50**，保存时必填（详见 5.3）|
| 坑位列表 | 按 POS 顺序排列的坑位集合，长度 = slotCount |
| Session 内去重 | 全局策略，系统默认开启，不可关闭 |
| 创建人 / 创建时间 | 用于权限判定（编辑权限仅创建人，超管可越权编辑/删除）与展示 |

**坑位（Slot）**

| 字段 | 业务含义 |
|---|---|
| 排序策略 | 该坑位绑定的排序策略，保存时必填 |

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **坑位顺序即 POS 序号**：数组第 i 项 → POS i，无独立 ID 暴露给外部
- **坑位粒度的标识不对外暴露**：装修系统只引用组合 ID，按返回数组顺序定位坑位，不引用单个坑位 ID

</callout>


### 5.3 坑位数（slotCount）

策略组合的坑位数由运营在创建时自定义，控制推荐结果的展示位数。

| 业务属性 | 说明 |
|---|---|
| 取值范围 | **1 ~ 50** 之间的整数 |
| 修改约束 | 仅在新建组合时填写，**保存后不可在原组合内修改** |
| 与资源位关系 | 本后台不感知装修资源位；装修服务负责把"资源位 → 组合 ID"映射对齐到合适的 slotCount |
| 缺省值 | 无默认值，运营必须显式填入 |

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **坑位数运营自决**：运营根据装修团队提供的资源位坑位需求自行决定，建议在创建前与装修对齐
- **保存后锁定**：保存后不可修改 slotCount，避免装修系统已对接的引用被静默改动；如需调整，请通过「复制」生成新组合后再改
- **不依赖资源位**：本后台不维护任何资源位元数据，组合列表 / 编辑页 / 接口均不出现资源位 ID 或名称
- **坑位顺序即展示顺序**：装修服务按返回数组顺序套到自身资源位的各位置上

</callout>


### 5.4 组合 ID

- 组合 ID 由系统在**创建时自动分配**，全局唯一
- **创建后不可变**：即使后续编辑组合的其他字段，ID 不变
- **删除后不复用**：删除某个 ID 后，该 ID 不再被新建占用
- **稳定引用**：装修配置以组合 ID 作为引用键，因此 ID 的稳定性是契约的一部分

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **ID 格式**：`comb-{自增数字}`（如 `comb-1` / `comb-7`），前缀由后端约定，自增数字部分由序列分配
- **ID 生成机制**：由后端单调递增分配（推荐用序列 / 全局计数器实现），不依赖当前表存量
- **ID 在 UI 中可复制**：编辑页基本信息区块明示展示组合 ID，并提供复制按钮，方便运营粘贴到装修配置

</callout>


### 5.5 同一策略复用与接续

- 同一排序策略可以在多个坑位中使用
- 后续坑位自动"接续"前序坑位的结果，展示该策略的**下一个排名商品**
- 界面用左侧色条 + 接续说明标记同一策略的多个坑位（如「接续 POS 1，展示第 2 名」）
- 同策略坑位的颜色仅在该策略出现 ≥2 次时分配，单次使用不上色

### 5.6 全局策略

- **Session 内去重**（sessionDedup）
  - 系统默认开启，不可关闭
  - 同一会话中已推荐过的商品不会再次推荐
  - 跨所有坑位、所有策略全局去重
  - **与 4.7 兜底链中"跨策略全局去重"是同一机制**：兜底取候选时也会跳过已展示商品，不重复推荐

### 5.7 覆盖商品数估算

系统根据组合中各坑位绑定的排序策略及其候选池，自动估算该组合可覆盖的商品总数，帮助运营了解推荐范围。

### 5.8 操作能力

#### 5.8.1 新建组合

**入口**：策略组合列表页右上角「新建策略组合」按钮。

**流程**：
1. 点击「新建策略组合」→ 系统分配自增 ID（如 `comb-7`），自动跳转到编辑页 `/combinations/{id}`，并以**编辑态**进入
2. 运营在编辑页填写：
   - 组合名称（必填）
   - 坑位数（必填，1-50 整数）→ 输入后 slots 数组自动初始化为对应数量的空策略坑位
   - 为每个 POS 选择排序策略（保存校验时必填）
3. 点击「保存」→ 校验通过后正式入库，跳回列表页

<callout emoji="📌" background-color="light-yellow">

**【关键操作规则】**

- **新建取消回滚**：在新建态点击「取消」或左上角返回箭头，**丢弃新建草稿，不留脏数据**
- **新建态 = 编辑态**：与"查看已有组合再切换到编辑"区分，新建态首次跳转就处于编辑模式
- **草稿对用户感知一致**：无论采用「立即写入 + 取消回滚」还是「前端本地草稿、保存时一次性提交」，对运营而言取消即抹掉，体验等价

</callout>


#### 5.8.2 查看与编辑切换

**默认模式**：所有用户从列表页点击「查看」进入编辑页，**默认进入只读态**。

**编辑权限**：仅当组合的「创建人 = 当前用户」**或当前用户是超管**时显示可点击的「编辑」按钮；其他用户「编辑」按钮置灰禁用。

**模式切换**：
- 只读态：header 显示「编辑 / 停用 / 删除」三个按钮（非创建者「编辑」+「停用」按钮置灰、隐藏「删除」），面包屑末项 = `详情`
- 编辑态：header 显示「取消 / 保存」按钮，面包屑末项 = `编辑`
- 点击「编辑」→ 切换到编辑态
- 点击「取消」→ 恢复 draft 到最新数据，回到只读态

<callout emoji="📌" background-color="light-yellow">

**【关键操作规则】**

- **编辑权限仅限创建人**：他人即使有"管理员"角色，也不能直接编辑别人的组合（如需修改，由创建人代操作或后续做转交功能）
- **取消语义因场景不同**：
  - 新建态取消 → 回滚草稿记录
  - 编辑已有组合取消 → 仅丢弃本地修改，组合本身保留

</callout>


#### 5.8.3 保存校验

按顺序校验，任一失败弹出 `保存失败` 提示并阻止保存：

| 序号 | 校验项 | 失败提示 |
|---|---|---|
| 1 | 组合名称非空（去首尾空白后 ≥1 字） | `策略组合名称不能为空` |
| 2 | 坑位数已填写且为 1-50 整数 | `请填写坑位数（1-50）` |
| 3 | 所有坑位的排序策略均已选（且策略状态为启用） | `请为 POS X、POS Y 选择策略`（动态列出缺失编号）/ `POS X 关联的策略已停用，请重新选择` |

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **三层校验不可绕过**：保存前必须全部通过，避免脏数据流入装修系统
- **POS 缺失提示具体到序号**：方便运营快速定位

</callout>


#### 5.8.4 启用 / 停用

- 列表页或编辑页头部操作；切换组合的状态
- 停用后，装修系统在调用时**走兜底**（详见 6.1 与装修的衔接说明）

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **停用不影响装修引用关系**：装修配置仍可保持原引用，运营随时启用即可恢复
- **列表页与详情页的启用/停用、删除均遵循创建人权限**：非创建者在两处入口均不可触发以上操作，按钮统一置灰或隐藏

</callout>


#### 5.8.5 删除

- **必须先停用才能删除**：组合状态为「启用」时「删除」按钮置灰禁用，悬停 Tooltip 提示「请先停用后再删除」
- 删除时弹二次确认，内容明示组合名 + 「该 ID 不会被新建复用，可能造成装修配置失效，请确认」

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **先停用再删除（硬约束）**：避免在线状态下被误删导致装修引用失效；启用态点击删除直接拒绝，不进二次确认
- **删除即让装修引用失效**：装修系统按 ID 调用时若组合不存在，按"走兜底"处理
- **删除后不复用 ID**：见 5.4

</callout>


#### 5.8.6 复制

- 列表页「复制」操作：以当前组合为模板创建新组合
- 复制规则：
  - 重新分配自增 ID
  - 名称加后缀「副本」（若超出 30 字上限则截断）
  - 创建人 = 当前用户
  - 创建时间 = 当前时间
  - 状态 = 停用（避免新副本立刻参与投放）
  - 坑位数（slotCount）、坑位策略全量克隆；如需修改 slotCount，正是通过"复制"再调整的标准路径

### 5.9 列表页交互

**表格列**（6 列）：

| 列 | 宽度 | 说明 |
|---|---|---|
| 名称 | 自适应 | 上行加粗中文名、下行组合 ID（次要字号）|
| 状态 | 100 | 启用 / 停用 标签 |
| 坑位数 | 100 | 数字右对齐，展示组合定义的 slotCount（1-50）|

**分页**：每页 12 条，底部显示总数。

## 6. 与装修系统的衔接

> 本节描述**本后台与装修系统的边界**。投放计划层（优先级 / 冲突检测 / 投放时间 / 门店人群范围 / ABTest）已**迁移至装修系统**，本后台不再承载。

### 6.1 用户视角的请求时序

完整推荐请求链路（按时序）：

1. 用户通过小程序进入点单页，前端发起推荐请求
2. **小程序请求装修服务**
3. **装修服务判断用户是否命中投放计划**（时间、门店、人群、ABTest 等命中条件由装修判定）
   - 未命中 → 装修自行兜底（不调推荐服务）
   - 命中 → 进入第 4 步
4. **装修服务携带以下信息调用推荐服务（即本后台出位接口）**：
   - (a) 用户 ID + 柜员 ID + 门店 ID
   - (b) **当前门店的可销售商品名称及商品 Item ID 列表**（装修上游已查好，推荐服务不重复查；**若未传入则推荐服务在选品池范围内兜底排序**）
   - (c) 命中计划所配置的**策略组合 ID**（资源位 → 组合 ID 映射由装修侧维护，本后台无感）
5. **推荐服务依据策略组合 ID 计算排序结果**（按组合定义的坑位数与顺序逐位执行排序策略 + 兜底链 + Session 去重）
6. **推荐服务返回结果给装修服务（或直接返回小程序）**：每位的商品名称 / Item ID / 来源标识 + 当前生效的组合元信息（ID、名称、状态）
7. 装修服务把推荐结果按返回顺序套到自身资源位的对应位置上，刷新到小程序展示

```text
小程序 ─→ 装修服务 ─(命中判定 + 资源位→组合ID 映射)─→ 推荐服务
                                       │       │
                                       │       │ ① 用户ID + 柜员ID + 门店ID
                                       │       │ ② 门店可售商品名称 + Item ID（可选）
                                       │       │ ③ 策略组合 ID
                                       │       ▼
                                       │   按 slotCount 计算排序结果
                                       │       │
                                       ◀───────┘
                                       │
                                       ▼
                                    小程序展示
```

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】 — 衔接模型**

- **装修服务是唯一调用方**：本后台不直接对接小程序；命中判定全部在装修服务侧
- **本后台不感知"何时、何地、对谁"**：时间 / 门店人群 / ABTest 由装修服务在调用前完成判定，本后台不接收这些参数
- **本后台不感知资源位**：装修系统在自己侧维护「资源位 → 组合 ID」映射；本后台仅按组合 ID 接收请求、按 slotCount 返回结果
- **门店可售商品由装修上游传入**：装修上游应过滤好"该门店当前在售"的商品名称 + Item ID 传给推荐服务；**若未传入，推荐服务回退到在组合所引用的选品池范围内排序**（防御性兜底）
- **组合是黑盒契约**：装修系统只看见组合 ID 与产出商品；不感知策略、池、加权等内部逻辑；**来源标识仅用于排障与埋点上报，不作为装修配置的依赖项**
- **组合不存在 → 错误码**；**组合存在但停用 → 空商品列表 + INACTIVE 标识**：两种情形装修均按"无可用组合"兜底

</callout>


### 6.2 出位接口约定（高层）

- **输入**：
  - 组合 ID（必填）：唯一入参，本后台按 ID 路由
  - 用户 ID + 柜员 ID + 门店 ID：用于排序策略取数 / 个性化计算，**不用于命中判定**
  - 门店可售商品列表（可选）：每项含商品名称 + Item ID；**若提供，排序在该列表内进行；若未提供，回退至组合所引用的选品池作为候选范围**
- **输出**：按 POS 顺序的结果列表（长度 = 组合.slotCount），每个坑位含：
  - 商品名称
  - 商品 Item ID（来自装修上游；若装修未传商品列表则来自选品池）
  - 来源标识（主策略 / 兜底 / 空置）
- **响应包含组合元信息**：组合 ID + 名称 + 状态（status: ACTIVE / INACTIVE）+ slotCount

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】 — 出位接口**

- **门店可售清单为权威候选（若提供）**：排序候选集 = 策略产出商品名称 ∩ 装修上游传入的门店可售商品名称；策略产出商品若不在该列表内，自动跳过取下一个
- **未传商品列表的兜底**：装修未传 (b) 时，推荐服务退回到"在组合各坑位所引用的选品池范围内"做排序，保证最低可用性
- **组合状态分类响应**：
  - 组合 ID 不存在 → HTTP 错误码（如 404）+ missReason，装修按"无可用组合"兜底
  - 组合存在但 status=INACTIVE → 返回 status=INACTIVE + 空商品列表，装修按"无可用组合"兜底
  - 组合存在且 status=ACTIVE → 正常返回排序结果
- **接口字段细节由研发约定**：错误码、SLA、缓存策略本 PRD 不展开

</callout>

## 7. 全链路预览

### 7.1 功能说明

预览**指定组合在指定上下文下的产出结果**，帮助运营验证策略组合的配置是否符合预期。

> 旧版"全链路预览"包含投放计划命中逻辑（时间 / 门店 / 人群 / ABTest 判定）。投放计划层已迁移至装修系统后，本预览**只模拟到组合产出**这一步，不模拟装修的命中环节。

### 7.2 输入参数

- **策略组合**：选择一个已创建的组合（或填入组合 ID）
- **门店**：选择模拟的门店（用于 HOT 模式数据范围）
- **用户 ID**：输入模拟用户 ID（用于 ALGORITHM 模式特征）
- **门店可售商品列表**：与 6.2 出位接口一致，由数据团队按所选门店实时拉取并预填；运营可手动剔除以模拟特殊场景。**若该参数为空，预览自动回退为"全量商品"模式并在面板顶部给出黄色警示**（提醒"模拟结果可能与实际出位不一致"）

### 7.3 输出结果

**组合产出面板**：
- 组合名 + 组合 ID + 坑位数（slotCount）+ 状态
- 每个 POS 的：策略名、模式、商品名、排名、来源（主策略 / 兜底 / 空置）

**策略链诊断面板**（按 POS 展开）：
- 主策略候选数 → 品类过滤后数量 → 截取数 → 加权前 / 后排名对比 → 兜底补足数
- Session 去重命中数
- **策略 / 池停用状态提示**：当前坑位的策略或其关联池是否被停用
- **NEW 模式聚合时间**：取到的"最新上架时间"具体值
- **兜底触发原因**：主策略候选不足 / 全部下架 / 策略停用 等
- 数据源对账（数仓表 N / 用户特征拉取耗时等占位提示）

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **预览不产生真实曝光**：仅本地模拟，不写入埋点和数据报告
- **使用 POS 序号表达坑位**：不暴露 slot.id
- **若组合停用，预览页给出明显警示**：但仍允许预览以辅助调试

</callout>

## 8. 效果监控

### 8.1 功能说明

实时查看推荐系统在各**策略组合**维度的核心指标，帮助运营评估推荐效果并优化策略。

> 监控维度由"投放计划"切换为"**策略组合**"。资源位维度（如有需要）由装修系统侧的数据看板覆盖；ABTest 与门店人群维度同样由装修系统看板提供，本系统不复述。

### 8.2 核心指标

| 指标 | 说明 |
|---|---|
| **CTR**（点击率）| 推荐位点击次数 / 曝光次数 |
| **CVR**（转化率）| 推荐位引导的下单次数 / 点击次数 |
| **归因 GMV** | 曝光量 × 客单价估算 |
| **坑位空置率** | 策略链未取到商品 + 兜底也未中的 POS 占比 |

### 8.3 分析能力

- **时间筛选**：自定义时间区间
- **维度筛选**：按"策略组合"筛选
- **效果 Top 3**：按曝光量排序的策略组合表现
- **趋势图**：CTR / CVR / 曝光量的时间趋势，支持切换指标

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **空置率口径**：坑位数由组合自定义（1-50），"空置"等价于"策略链未取到商品"
- **不再按投放计划分组**：相关需求请前往装修系统的数据看板
- **资源位维度不在本后台监控**：资源位归装修管理，相关报表请查阅装修系统看板

</callout>


### 8.4 数据回传口径（与装修系统约定）

监控指标必须依赖装修系统上报的埋点事件，事件必须携带以下字段才能在本后台按"策略组合"维度聚合：

| 必传字段 | 用途 |
|---|---|
| `combinationId` | 组合归属（监控聚合主键）|
| `pos` | 坑位序号（POS 1 / 2 / ...）|
| `productName` | 商品名称（推荐对象的稳定标识）|
| `source` | 来源标识：主策略 / 兜底 / 空置 |

> 装修系统侧的"资源位归属"由装修在自己上报通道中维护，与本后台监控无关。

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】 — 数据回传**

- **缺字段 = 无法纳入监控**：装修上报的曝光 / 点击 / 订单归因事件若缺少上述任一字段，本后台不能按组合维度监控，需运营推动装修补齐
- **来源标识仅用于监控分组**：本后台依据 `source` 区分主策略命中、兜底命中和空置三类，分别计算指标；装修系统不依赖此字段做配置

</callout>

## 9. 权限控制

### 9.1 设计原则

- **粗粒度权限**：本期只做 12 个 `resource:read|write` 权限点（移除原 `plan:*` 两个），不做按钮 / 字段级细化
- **角色 kind**：仅"超级管理员"是 `SYSTEM`（硬编码不可改、不可删），其余角色全部 `CUSTOM`
- **多角色**：一个用户可分配多个角色，最终权限取所选角色权限的**并集**
- **角色名称唯一**：新建 / 编辑角色时强制校验名称不重复
- **不做 DataScope**：本期不做"数据范围"过滤（按区域、按门店等）
- **读写分级生效**：仅有 `resource:read` 时，对应模块的列表 / 详情页可访问，但所有"新建 / 编辑 / 停用 / 删除"按钮统一置灰禁用；无 `resource:read` 时，对应模块在侧边栏隐藏，直接访问路由跳转 403

### 9.2 角色清单（初始预置 4 档，除超管外都可改可删）

| 角色 | kind | 权限范围 |
|---|---|---|
| **超级管理员** | `SYSTEM`（硬编码）| 12 项全开（含 user / role）|
| 管理员 | `CUSTOM` | 业务全开，不含 user / role |
| 小程序配置 | `CUSTOM` | strategy / combination RW，pool 只读，monitoring / preview 只读 |
| 商品运营 | `CUSTOM` | pool RW，其他业务资源只读 |

### 9.3 权限点（12 个粗粒度 `resource:action` 串）

| 资源 | 权限点 |
|---|---|
| 选品池 | `pool:read` / `pool:write` |
| 排序策略 | `strategy:read` / `strategy:write` |
| 策略组合 | `combination:read` / `combination:write` |
| 效果监控 | `monitoring:read` |
| 全链路预览 | `preview:read` |
| 用户管理 | `user:read` / `user:write` |
| 角色管理 | `role:read` / `role:write` |

<callout emoji="📌" background-color="light-yellow">

**【关键业务规则】**

- **粗粒度即放即用**：`combination:write` 即可编辑/启停/删除组合；后续若要拆细再单独开
- **业务对象的"创建人 = 编辑权"约束独立于角色**：见 5.8.2，组合的编辑权一般只给创建人，不被普通角色覆盖
- **超管例外**：超级管理员可越权编辑 / 停用 / 删除任何用户创建的池 / 策略 / 组合（即"创建人 = 编辑权"对超管不生效）；其余 CUSTOM 角色即便拥有 `xxx:write` 权限，也只能管理自己创建的数据
- **plan 相关权限已删除**：本后台不再有投放计划相关入口

</callout>


### 9.4 用户来源（飞连同步）

- **用户来源**：直接对接公司**飞连员工目录**，不在后台单独建账号、不维护密码体系
- **新建用户**：唯一手填字段是"用户姓名"，输入后做联想搜索；命中后自动带出工号 / 账号 / 手机 / 邮箱 / 职位 / 部门 / 直接上级（7 项只读）
- **后台只配置**：角色（多选）、备注、状态（正常 / 已禁用）
- **重复校验**：同一名飞连员工不允许重复添加；重复时引导跳到该账号编辑

## 10. 附录

### 10.1 商品定义与品类规则

- **推荐对象**：商品名称（非 SKU / 非 SPU）
- **底层去重键**：SPU ID（仅用于数仓 / 校验，不暴露给运营 UI 或推荐输出）
- **数仓品类**：饮品 / 套餐 / 小料 等
- **品类约束**：
  - 选品池仅展示饮品；其他品类不进入候选
  - HOT / NEW / ALGORITHM 策略仅消费饮品
  - **MANUAL 策略可跨品类选商品**（饮品 + 套餐 + 小料）

### 10.2 资源位（已迁出至装修系统）

> 自 v3.4 起，本后台不再维护"资源位"概念：组合不再绑定资源位字段，列表 / 编辑 / 接口均不展示资源位 ID 或中文名。
>
> 装修系统在自己侧维护「资源位 → 组合 ID」映射，本后台只对外提供"按组合 ID 取推荐结果（按 slotCount 返回）"。
>
> 运营如需对齐"哪个资源位用哪个组合"，请向装修团队咨询装修配置。

### 10.3 排序策略示例

| 策略名称 | 模式 | 关联池 | 说明 |
|---|---|---|---|
| 全量热销榜 | HOT | ALL 全量池 | 系统内置兜底策略，数据范围 = 门店级，时间窗口 = 7 天 |
| 果茶人工定序 | MANUAL | —（不绑池）| 活动期手动排序，可加套餐 |
| 咖啡早高峰 | HOT | 咖啡池 | 数据范围 = 门店，时间窗口 = 7 天 |
| 新品推广 | NEW | 上新池 | 时间窗口 = 14 天 |
